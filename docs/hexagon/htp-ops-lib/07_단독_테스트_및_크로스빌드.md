## 1. 단독 테스트 가능 여부: **Yes**

htp-ops-lib는 llama.cpp 본체 없이 **완전히 독립적으로** 빌드 및 테스트할 수 있도록 설계되어 있다.

### 1.1 Stub/Skel 구조로 완전 독립 빌드

> 코드 위치: [CMakeLists.txt](../../../../htp-ops-lib/CMakeLists.txt)

빌드 시 두 개의 독립 바이너리가 생성된다:

| 빌드 대상 | 아키텍처 | 산출물 | 역할 |
|-----------|---------|--------|------|
| HLOS (Host) | AArch64 (Android CPU) | `libhtp_ops.so` (stub) | FastRPC 호출 인터페이스 |
| HLOS (Host) | AArch64 (Android CPU) | **`htp_ops_test`** (실행 파일) | 독립 테스트 프로그램 |
| Hexagon (DSP) | Q6DSP | `libhtp_ops_skel.so` (skel) | DSP 측 실제 연산 구현 |

`htp_ops_test`라는 **독립 테스트 실행 파일**이 HLOS 빌드에 포함되어 있어, llama.cpp 없이 DSP 연산을 직접 호출하고 속도를 측정할 수 있다.

## 2. 테스트 프로그램 구조

### 2.1 호스트 측 테스트 (Android CPU에서 실행)

> 코드 위치: [src/host/test.c](../../../../htp-ops-lib/src/host/test.c)

| 함수 | 테스트 내용 | 측정 항목 |
|------|-----------|-----------|
| `test_rms_norm_f32_rpc()` | FastRPC 경로로 RMS Norm 호출 + CPU 레퍼런스 비교 | **RPC 레이턴시 (μs)** |
| `test_rms_norm_f32_chan()` | 공유 메모리 채널 경로로 동일 테스트 | **채널 레이턴시 (μs)** |
| `test_mat_mul_rpc()` | MatMul (FP16 weight) RPC 호출 + Crouton 인덱싱 검증 | 수치 정확도 비교 |
| `htp_ops_test_ops()` | DSP 측 벤치마크 트리거 (IDL의 `test_ops()` RPC 호출) | DSP 측 GFLOPS/대역폭 |

### 2.2 DSP 측 벤치마크 (Hexagon NPU에서 실행)

> 코드 위치: [src/dsp/op_tests.cc](../../../../htp-ops-lib/src/dsp/op_tests.cc)

| 함수 | 벤치마크 내용 | 측정 항목 |
|------|-------------|-----------|
| `benchmark_hmx_gemm()` | HMX FP16 타일 기반 GEMM (32~1024 크기, 1000회 반복) | **GFLOPS** |
| `benchmark_hvx_gemm()` | HVX FP16 GEMM (1~4 스레드, 10회 반복) | **GFLOPS** |
| `benchmark_vtcm_bandwidth()` | VTCM 읽기/쓰기 대역폭 (HVX 벡터 복사) | **GB/s** |
| `test_fp16_exp2()` | HVX exp2 다항식 근사 정확도 검증 | 수치 비교 |
| `test_int16_fp16_conversion()` | INT16↔FP16 변환 테스트 | 수치 비교 |

### 2.3 IDL 인터페이스

> 코드 위치: [include/htp_ops.idl](../../../../htp-ops-lib/include/htp_ops.idl)

```idl
interface htp_ops : remote_handle64 {
    AEEResult init_backend();
    AEEResult create_channel(in int32 fd, in uint32 size);
    AEEResult destroy_channel();

    /* 연산자 RPC 인터페이스 */
    AEEResult rms_norm_f32(...);
    AEEResult mat_mul_permuted_w16a32(...);
    
    AEEResult test_ops();    // ← DSP 측 벤치마크 트리거
};
```

## 3. 테스트 실행 흐름

```
htp_ops_test (Android CPU에서 실행)
  │
  ├─ open_dsp_session()              // DSP 세션 열기
  ├─ init_htp_backend()              // HMX/VTCM/Power 초기화
  │
  ├─ htp_ops_test_ops()              // DSP에서 벤치마크 실행
  │     └─▶ FastRPC ─▶ internal_op_tests()  (DSP에서 실행)
  │           ├─ benchmark_hmx_gemm()        → GFLOPS 출력
  │           ├─ benchmark_hvx_gemm()        → GFLOPS 출력
  │           └─ benchmark_vtcm_bandwidth()  → GB/s 출력
  │
  ├─ test_rms_norm_f32_rpc()         // RPC 레이턴시 + 정확도
  ├─ test_mat_mul_rpc()              // MatMul 정확도
  └─ close_dsp_session()
```

> **주의**: DSP 측 벤치마크 함수들은 현재 [op_tests.cc의 `internal_op_tests()` 함수](../../../../htp-ops-lib/src/dsp/op_tests.cc) 내에서 주석 처리되어 있다. 원하는 벤치마크의 주석을 해제한 후 다시 빌드해야 한다.

## 4. 크로스빌드 방법

### 4.1 사전 준비

- **Hexagon SDK 6.x** 설치 (검증 버전: 6.0.0.2)
- SDK 내 `build_cmake` 스크립트와 `hexagon_fun.cmake` 필요

### 4.2 빌드 순서

```bash
# 1. Hexagon SDK 환경 설정 (SDK 루트 디렉토리에서)
cd $HEXAGON_SDK_ROOT
source setup_sdk_env.source

# 2. htp-ops-lib 디렉토리로 이동
cd /path/to/llama.cpp/htp-ops-lib

# 3. Android(AArch64) Stub + 테스트 실행 파일 빌드
build_cmake android

# 4. Hexagon(DSP) Skel 빌드
build_cmake hexagon DSP_ARCH=v73
```

### 4.3 빌드 결과물

```
htp-ops-lib/
├── android_ReleaseG_aarch64/
│   ├── libhtp_ops.so          ← Stub (Android CPU용)
│   └── htp_ops_test           ← 독립 테스트 실행 파일
└── hexagon_ReleaseG_toolv87_v73/
    └── libhtp_ops_skel.so     ← Skel (Hexagon DSP용)
```

### 4.4 빌드 시스템 상세

> 코드 위치: [CMakeLists.txt](../../../../htp-ops-lib/CMakeLists.txt)

```cmake
# Hexagon SDK의 CMake 헬퍼 포함
include(${HEXAGON_SDK_ROOT}/build/cmake/hexagon_fun.cmake)

if(${OS_TYPE} MATCHES "HLOS")
    # --- Android(AArch64) 빌드 ---
    add_library(htp_ops SHARED ...)        # Stub 라이브러리
    add_executable(htp_ops_test ...)       # 테스트 실행 파일
    build_idl(include/htp_ops.idl htp_ops) # QAIC IDL 코드 생성
else()
    # --- Hexagon(Q6DSP) 빌드 ---
    add_library(htp_ops_skel SHARED ...)   # Skel 라이브러리
    set(CMAKE_C_FLAGS "${CMAKE_C_FLAGS} -mhmx")  # HMX 인라인 어셈블리 활성화
endif()
```

### 4.5 DSP_ARCH 옵션

| 값 | 대상 SoC | 비고 |
|----|---------|------|
| `v73` | Snapdragon 8 Gen 2 | **기본 권장**, 호환성 좋음 |
| `v75` | Snapdragon 8 Gen 3 | v73 호환 가능 |
| `v79` | 최신 SoC | ⚠ 일부 FP 연산 오류 가능 (Known Issue) |

> **Known Issue**: DSP Arch v79에서 높은 버전의 Hexagon SDK로 빌드 시 일부 부동소수점 계산 오류 발생 가능. v79 대상 하드웨어에서는 v73/v75로 빌드 후, `flash_attn.c` 내 `enable_vgather_exp`를 `false`로 설정 권장.

## 5. 디바이스 배포 및 실행

### 5.1 ADB를 통한 파일 전송

```bash
# so 파일 및 테스트 실행 파일 푸시
adb push android_ReleaseG_aarch64/libhtp_ops.so /data/local/tmp/
adb push hexagon_ReleaseG_toolv87_v73/libhtp_ops_skel.so /data/local/tmp/
adb push android_ReleaseG_aarch64/htp_ops_test /data/local/tmp/
```

### 5.2 테스트 실행

```bash
# DSP 라이브러리 검색 경로 설정 + 실행
adb shell "export ADSP_LIBRARY_PATH=/data/local/tmp && \
           chmod +x /data/local/tmp/htp_ops_test && \
           /data/local/tmp/htp_ops_test"
```

### 5.3 예상 출력 예시

```
rms_norm_f32 RPC took 150 us
test_rms_norm_f32_rpc passed
core fp16 hmx: 1234.56 GFLOPS@n=1024, 1650 us
core fp16 hvx: 12.34 GFLOPS@1 Threads, 165000 us
vtcm bandwidth: 45 us, read bw: 91.02 GB/s, total bw: 182.04 GB/s
```

## 6. 요약

| 질문 | 답변 |
|------|------|
| 단독 테스트 가능? | **Yes** — `htp_ops_test` 실행 파일이 빌드됨 |

| llama.cpp 의존성? | **없음** — 완전 독립 빌드 |
| 크로스빌드? | Hexagon SDK의 `build_cmake android` + `build_cmake hexagon DSP_ARCH=v73` |
